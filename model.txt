================================================================================
  1.  ARCHITECTURE OVERVIEW
================================================================================

Acadomi is a three-tier system:

  ┌──────────┐      ┌──────────────────┐      ┌──────────────────────────────┐
  │ Frontend │ ←──→ │  Node.js/Express │ ←──→ │  Python Microservices (x2)   │
  │ (Next.js)│      │   Backend API    │      │  • Podcast (Flask, port 5001)│
  └──────────┘      │   (port 4000)    │      │  • Tutor  (FastAPI, port 5002│
                    └──────────────────┘      └──────────────────────────────┘
                            ↕                          ↕
                      ┌──────────┐            ┌────────────────┐
                      │ MongoDB  │            │ External APIs: │
                      │ + GridFS │            │ • Google Gemini│
                      └──────────┘            │ • Ollama (local│
                                              │ • Edge TTS     │
                                              │ • gTTS         │
                                              └────────────────┘

All LLM calls from Node.js flow through a single router:
  backend/src/services/llm/router.ts

That router checks the LLM_BACKEND env var and dispatches to either:
  • Google Gemini  (@google/generative-ai SDK)
  • Ollama (local) (HTTP fetch to Ollama REST API)

The two Python services have their OWN model usage:
  • Podcast service calls Gemini directly (google.generativeai Python SDK)
  • Tutor service does NOT call any LLM — it runs Edge TTS and MediaPipe only


================================================================================
  2.  LLM ROUTER  (backend/src/services/llm/)
================================================================================

The LLM layer consists of 4 files:

  FILE                    PURPOSE
  ─────────────────────── ──────────────────────────────────────────────────────
  config.ts               Reads env vars; exports backend type & model names
  ollama.ts               HTTP client for Ollama /api/chat (text & vision)
  router.ts               Central router: decides Gemini vs Ollama per call
  llmReady.ts             Guards: isLlmConfigured() and hasGeminiForTranscription()

─── config.ts ───

  getLlmBackend()           → "gemini" | "ollama"
                             Reads LLM_BACKEND env. Default: "gemini".
                             If "auto": uses ollama when OLLAMA_HOST is set.

  ollamaBaseUrl()           → default http://127.0.0.1:11434

  ollamaModelTutor()        → env OLLAMA_MODEL_TUTOR   or "llama2"
  ollamaModelLight()        → env OLLAMA_MODEL_LIGHT   or "phi3:mini"
  ollamaModelStructured()   → env OLLAMA_MODEL_STRUCTURED or "phi3"
  ollamaModelVision()       → env OLLAMA_MODEL_VISION  or "llava"

  getImageTextBackend()     → "gemini" | "tesseract" | "ollama"
                             Reads IMAGE_TEXT_BACKEND env. Defaults:
                             if LLM_BACKEND=ollama → "tesseract", else "gemini"

─── ollama.ts ───

  ollamaChat(model, prompt, system?)
      Single-turn, non-streaming chat via POST /api/chat.
      10-minute timeout. Used for text-only Ollama calls.

  ollamaVisionChat(model, prompt, imageBase64)
      Vision chat. Passes base64 image in the "images" array.
      Used for OCR-via-LLM when IMAGE_TEXT_BACKEND=ollama.

─── router.ts (THE CORE DISPATCHER) ───

  getGeminiModel()
      Reads GEMINI_API_KEY and GEMINI_MODEL (default "gemini-2.5-flash").
      Returns a GoogleGenerativeAI model instance.

  FUNCTION                    GEMINI PATH                    OLLAMA PATH
  ────────────────────────── ──────────────────────────────── ────────────────────
  completeTutorPrompt()       gemini-2.5-flash               llama2 (via Ollama)
  completeLightPrompt()       gemini-2.5-flash               phi3:mini (via Ollama)
  completeStructuredPrompt()  gemini-2.5-flash               phi3 (via Ollama)
  extractTextFromImage()      gemini-2.5-flash (multimodal)  llava (via Ollama)
                              OR tesseract.js (no LLM)
  transcribeAudioBuffer()     gemini-2.5-flash (multimodal)  ❌ NOT SUPPORTED
                              (audio only works with Gemini)

─── llmReady.ts ───

  isLlmConfigured()
      True if LLM_BACKEND=ollama OR GEMINI_API_KEY is set.
      All generation routes check this before proceeding.

  hasGeminiForTranscription()
      True only if GEMINI_API_KEY is set. Required for audio features
      (speech-to-text) since Ollama has no built-in audio path.

3.3  AI TUTOR — SLIDE GENERATION
────────────────────────────────────────────────────────────────────────────────

  Route:     POST /api/tutor/sessions
  File:      backend/src/routes/tutor.ts  (line 106)
  Function:  generateTutorSlidesAndScripts(material, topicFocus)
  File:      backend/src/services/gemini.ts  (line 377)
  LLM call:  completeTutorPrompt()

  MODEL (Gemini mode):    gemini-2.5-flash
  MODEL (Ollama mode):    llama2

  What it does:
    Converts study material into 6–14 structured slides. Each slide has:
      • title        — short heading
      • points[]     — 2–5 bullet points
      • script       — spoken narration text (40–90 words, conversational)

  Output format:  JSON  {"slides":[{"title":"...","points":[...],"script":"...","pointTimings":[...]}]}
  Input limit:    material sliced to 100,000 chars.

  Post-processing in normalizeTutorSlides():
    • Validates each slide has title, script, and ≥1 point
    • Falls back to word-count-weighted timing if model timings are invalid
    • Requires ≥3 valid slides; caps at 16 slides max

3.6  ROLE-REVERSAL EVALUATION
────────────────────────────────────────────────────────────────────────────────

  Route:     POST /api/role-reversal/evaluate
  File:      backend/src/routes/roleReversal.ts  (line 43)

  Pipeline (two LLM calls):
    1. transcribeAudio(audioBuffer)
       → Gemini 2.5 Flash (multimodal)

    2. evaluateRoleReversalTeaching({topic, referenceMaterial, studentTranscript})
       File: backend/src/services/gemini.ts  (line 164)
       → completeStructuredPrompt()

  MODEL for transcription:   Gemini 2.5 Flash (always)
  MODEL for evaluation:
    Gemini mode:  gemini-2.5-flash
    Ollama mode:  phi3

  What it does:
    The student records themselves "teaching" a topic. The audio is transcribed,
    then compared against reference material. The LLM outputs a structured JSON
    evaluation:

      {
        "scoreClarity":    0–100,
        "scoreConcepts":   0–100,
        "scoreFluency":    0–100,
        "totalScore":      0–100,
        "feedback":        "2–4 sentences",
        "topicUnderstanding": "...",
        "weakness":        "...",
        "strength":        "...",
        "visualHints": {
          "radar":      [{label, value}...],
          "barCompare": [{label, you, ideal}...]
        }
      }

  Post-processing in normalizeEvaluation():
    • Clamps all scores to 0–100
    • Calculates totalScore as average if model omits it
    • Normalizes camelCase / snake_case key variants

3.10  PODCAST SCRIPT GENERATION (Python)
────────────────────────────────────────────────────────────────────────────────

  Route:     POST /api/podcasts/generate  (Node backend proxies to Python)
  File:      backend/src/routes/podcasts.ts  (line 48)
             → HTTP POST to Python service at PODCAST_SERVICE_URL/generate-podcast

  Python file:  python/services/podcast/podcast_logic.py
  Function:     generate_script_from_gemini(source_text, ...)
  SDK:          google.generativeai (Python SDK — NOT the Node SDK)

  MODEL:   Gemini 2.5 Flash  (configurable via GEMINI_MODEL env)
           ⚠ ALWAYS uses Gemini. No Ollama path in the podcast service.

  What it does:
    Converts study material into a two-person conversational podcast script
    between "Alice" and "Bob". Output is a JSON array:
      [{"speaker":"Alice","text":"..."},{"speaker":"Bob","text":"..."}, ...]

    Rules enforced via prompt:
      • 8–14 short turns
      • Faithful to source material
      • Plain English suitable for text-to-speech
      • Alternating speakers

  Post-processing in _parse_dialogue_json_array():
    • Strips markdown code fences
    • Handles wrapped objects {"dialogue": [...]}
    • Falls back to greedy bracket regex matching
    • Validates speaker names are exactly "Alice" or "Bob"

  The GEMINI_API_KEY is forwarded from the Node backend in the request body.
  This allows the Python service to run without its own .env when called by Node.

3.11  PODCAST AUDIO SYNTHESIS (Python)
────────────────────────────────────────────────────────────────────────────────

  Python file:  python/services/podcast/podcast_logic.py
  Function:     create_podcast_mp3_bytes(script)
  Libraries:    gTTS (Google Text-to-Speech) + pydub + FFmpeg

  MODEL:   gTTS (Google Cloud TTS — free tier, no API key needed)
           NOT an LLM. This is a neural TTS service.

  What it does:
    Takes the dialogue script from 3.10 and synthesizes audio:
      • Alice → British English accent (tld="co.uk")
      • Bob   → Indian English accent (tld="co.in")
      • 400ms silence between turns
      • Combined into single MP3 at 64kbps
    Returns base64-encoded MP3 bytes + duration.

  Dependencies:
    • gTTS — calls Google Translate's TTS endpoint (no API key)
    • pydub — audio segment manipulation
    • FFmpeg/FFprobe — required by pydub for MP3 encoding/decoding
      (extensive auto-discovery logic for Windows WinGet installs)

  The audio is stored in MongoDB GridFS via:
    backend/src/services/gridfsPodcast.ts

3.13  FOCUS / ATTENTION MONITORING (Python — Computer Vision)
────────────────────────────────────────────────────────────────────────────────

  Routes:
    POST /api/tutor/sessions/:id/focus/reset    (reset tracker for session)
    POST /api/tutor/sessions/:id/focus/analyze  (analyze one webcam frame)

  Node-side:  tutor.ts calls pyFocusReset() and pyFocusAnalyze()
              → HTTP to TUTOR_SERVICE_URL/focus/reset and /focus/analyze

  Python-side:
    File:      python/services/tutor/app.py       (FastAPI endpoints)
    File:      python/services/tutor/focus_tracker.py  (core logic)

  MODEL:   MediaPipe FaceMesh (Google)
           Version: mediapipe >=0.10.21, <0.10.30 (requires Python 3.10–3.12)
           NOT an LLM. This is a real-time face landmark detection model.

  Supporting library:  OpenCV (cv2) for image decoding and pose estimation

  What it does (FocusTracker class):

    CALIBRATION PHASE (first 30 frames):
      • Records baseline pitch, yaw, and EAR (Eye Aspect Ratio)
      • Determines the student's "natural" head position and eye openness

    MONITORING PHASE (every subsequent frame):
      a) HEAD POSE ESTIMATION
         Uses 6 facial landmarks + solvePnP to compute pitch/yaw/roll.
         Compares against baseline with configurable tolerances:
           PITCH_TOLERANCE = 25°    HEAD_DOWN_THRESH = 45°
           YAW_TOLERANCE   = 20°

      b) EYE ASPECT RATIO (EAR)
         Measures eye openness using 6 landmarks per eye.
         Detects:
           • Blinks    (EAR drops > 0.03 from baseline)
           • Drowsiness (sustained EAR drop > 0.017 for > 0.85s)
           • Eyes closed (EAR drop > 0.04 sustained > 15s → WAKE UP alarm)

      c) IRIS GAZE TRACKING
         Tracks iris positions (landmarks 468, 473) relative to eye corners.
         Computes gaze variance over 30-frame window.
         Detects staring (no blinks for 15s + no gaze movement for 15s).

      d) COMPOSITE FOCUS SCORE (0–100)
         focus = pose_score × 0.4 + eye_score × 0.3 + gaze_score × 0.3
         Smoothed over 20-frame rolling history.

         STATUS LEVELS:
           > 75  → FOCUSED
           > 40  → DISTRACTED
           ≤ 40  → NOT FOCUSED (triggers alarm)

         ALARM CONDITIONS:
           • Eyes closed > 15s     → "WAKE UP"
           • Head down > 15s       → "HEAD DOWN"
           • Staring (no blink/gaze for 15s) → "PLEASE BLINK"

    Thread safety:
      FaceMesh must run on the same thread it was created on.
      All tracker work is funneled through a single-thread ThreadPoolExecutor.

4.1  PODCAST SERVICE  (python/services/podcast/)
────────────────────────────────────────────────────────────────────────────────

  Framework:   Flask
  Port:        5001 (FLASK_PORT env)
  Entry point: python/services/podcast/app.py

  FILES:
    app.py             — Flask HTTP server, /health and /generate-podcast
    podcast_logic.py   — All generation logic (Gemini script + gTTS audio)
    .env.example       — Configuration template
    requirements.txt   — (in _template/ sibling)

  ENDPOINTS:

    GET  /health
      Returns service status + ffmpeg availability check.

    POST /generate-podcast
      Body: { "text": "...", "geminiApiKey": "...", "geminiModel": "..." }
      Returns: { "script": [...], "mimeType": "audio/mpeg",
                 "durationMs": N, "audioBase64": "..." }

  CODE FLOW (build_podcast_payload):
    1. ensure_pydub_ffmpeg()
       → Validates ffmpeg/ffprobe are available
       → Searches: env vars → WinGet packages → PATH → shutil.which
       → Patches pydub's converter and prober paths

    2. generate_script_from_gemini(source_text)
       → Configures Gemini (genai.configure(api_key=...))
       → Creates GenerativeModel("gemini-2.5-flash")
       → Sends prompt asking for JSON dialogue array
       → Parses response with _parse_dialogue_json_array():
           a) Strips markdown fences
           b) raw_decode from first '['
           c) Falls back to {"dialogue": [...]} object wrapper
           d) Falls back to greedy regex [\s\S]*]
       → Validates: speaker ∈ {"Alice","Bob"}, text non-empty, ≥2 lines

    3. create_podcast_mp3_bytes(script)
       → For each dialogue turn:
           a) gTTS(text, lang="en", tld="co.uk"|"co.in")  → MP3 file
           b) AudioSegment.from_mp3(path)
           c) Append 400ms silence
       → Concatenate all segments
       → Export combined audio as MP3 at 64kbps
       → Return (bytes, duration_ms)
       → Clean up temp directory

================================================================================
  7.  END-TO-END DATA FLOW DIAGRAMS
================================================================================


─── UPLOAD → STUDY NOTES ───

  User uploads PDF/Image/Audio
       │
       ▼
  [Node] uploads.ts validates files
       │
       ├─ PDF  → pdf-parse (no LLM)
       ├─ Image → Gemini Vision / tesseract.js / llava
       └─ Audio → Gemini multimodal (audio→text)
       │
       ▼
  extractedText (raw content)
       │
       ▼
  synthesizeLearningNotes() → completeLightPrompt()
       │                       (Gemini 2.5 Flash or phi3:mini)
       ▼
  processedContent (markdown study notes)
       │
       ▼
  Saved to MongoDB (Upload document)


─── TUTOR SESSION LIFECYCLE ───

  User selects Upload → creates session
       │
       ▼
  generateTutorSlidesAndScripts() → completeTutorPrompt()
       │                           (Gemini 2.5 Flash or llama2)
       ▼
  6–14 slides with scripts + point timings
       │
       ▼
  Saved to MongoDB (TutorSession document)
       │
       ▼
  Frontend loads session → requests TTS per slide
       │
       ▼
  [Node] tutorPyTts(script) → [Python] Edge TTS → MP3 audio
       │
       ▼
  Audio plays in browser with synced bullet animations
       │
       ▼  (if student asks a question)
  [Node] transcribeAudio() → Gemini (audio→text)
       │
       ▼
  answerTutorQuestion() → completeTutorPrompt()
       │                   (Gemini 2.5 Flash or llama2)
       ▼
  Answer text → TTS → Audio response


─── PODCAST GENERATION ───

  User clicks "Generate Podcast" on an upload
       │
       ▼
  [Node] podcasts.ts → HTTP POST to Python podcast service
       │                (forwards GEMINI_API_KEY + material text)
       ▼
  [Python] generate_script_from_gemini()
       │   → Gemini 2.5 Flash (via google.generativeai SDK)
       │   → Returns JSON dialogue [{speaker, text}, ...]
       ▼
  [Python] create_podcast_mp3_bytes()
       │   → gTTS (Alice=co.uk, Bob=co.in) → individual MP3s
       │   → pydub concatenation + 400ms gaps → combined MP3
       ▼
  audioBase64 + script returned to Node
       │
       ▼
  [Node] stores MP3 in MongoDB GridFS
         stores metadata in Podcast collection


─── FOCUS MONITORING ───

  Webcam frame captured in browser (every ~1–2 seconds)
       │
       ▼
  [Node] tutor.ts → POST to Python /focus/analyze
       │             (image buffer as base64)
       ▼
  [Python] focus_tracker.py
       │   → cv2.imdecode → BGR frame
       │   → MediaPipe FaceMesh → 478 face landmarks
       │   → Head pose (solvePnP) + EAR + iris gaze
       │   → Composite focus score (0–100)
       ▼
  JSON response: {focusVal, status, alarm, ...}
       │
       ▼
  Frontend renders focus ring indicator
